01
The Problem

Context is scattered across too many tools.

🔀
Tool Switching Overhead
Every task requires switching between Jira, GitLab, Slack, Confluence, and the MariaDB console. Context gets lost at every boundary.
🧩
Fragmented Knowledge
Who owns NET-5451? What did we decide in that Slack thread last Tuesday? Which pipeline broke the deploy? The answers live in five different places.
⏳
Repetitive Lookups
Standup prep, ticket status checks, MR reviews, log searches — every developer does these manually, every single day.
📉
Slow Incident Response
When something breaks, correlating a failing pipeline with the relevant ticket, the Slack thread, and the gameserver logs takes too long.
02
The Solution

One AI. All the context. Connected.

🧠
Company Brain
Built as a Model Context Protocol server in Deno — plugged directly into Claude Code, which the team is already using. No new tool to adopt. No new subscription.
03
Capabilities

What it can do right now.

🎫
Jira
  • List, search, create and update tickets
  • Transition status as work progresses
  • Add comments, link issues, manage sprints
  • Read attachments and ticket history
🦊
GitLab
  • Browse projects, branches, files, trees
  • Create branches, commits, merge requests
  • Review MRs: diffs, comments, approvals
  • Trigger, monitor and retry CI pipelines
💬
Slack
  • Search messages and threads across all channels
  • Send messages and open DMs
  • Fetch files shared in channels
  • Resolve people by name without guessing IDs
📑
Confluence
  • Search pages by CQL query
  • Read full page content (any space)
  • Create and update pages
  • Browse spaces and page hierarchies
🗄️
MariaDB
  • Read-only access to production data
  • SELECT, SHOW, DESCRIBE, EXPLAIN
  • Investigate payment issues, server state
  • Answer customer data questions instantly
🔍
Elasticsearch
  • Full-text search across gameserver logs
  • Filter by time range, severity, service
  • Instant incident investigation without Kibana
  • Correlate log events with Jira tickets
📊
Loki
  • Query Fleet container logs via LogQL
  • Filter by app, container, host, environment
  • Tail v4 API and gameserver logs in context
  • Correlate log output with CI pipeline failures
🌐
Browser Automation
  • Navigate, click, fill forms in any web UI
  • Take screenshots and read page content
  • Automate tools that have no API
  • Research, scraping, visual verification
🤖
Workflows
  • Daily standup summaries in seconds
  • Ticket → branch → MR → transition, automated
  • Cross-tool incident correlation
  • "What should I work on next?" answered
04
Example Workflows

Two prompts. Full context. Zero tab-switching.

Workflow 01 — Dev
Pick up NET-5451 and open an MR.
1
Fetch the ticket
Reads description, acceptance criteria, status and linked issues. No copy-pasting.
Jira
2
Check related context
Searches Slack for recent discussion. Checks Confluence for architecture docs.
Slack · Confluence
3
Create branch + implement
Creates a GitLab branch named after the ticket ID, reads existing code, writes the fix.
GitLab
4
Open MR + move ticket
Creates a merge request with a proper description. Transitions Jira to "Feedback" automatically.
GitLab · Jira
5
Monitor CI
Watches the pipeline, fetches job logs on failure, suggests fixes.
GitLab CI
Workflow 02 — Ops
Why is gameserver #42 crashing since 14:00?
1
Search gameserver logs
Queries Elasticsearch for errors and stack traces around the crash window. Spots the pattern immediately.
Elasticsearch
2
Check container logs
Pulls Loki logs from the affected fleet pod to correlate OOM kills, restarts and sidecar errors.
Loki
3
Query affected sessions
Runs a read-only DB query to find impacted players, match counts and session durations in the crash window.
MariaDB
4
Look up runbook + notify
Finds the incident runbook in Confluence. Posts a summary with root cause and affected scope to the ops channel.
Confluence · Slack
5
Open monitoring dashboard
Launches the browser, navigates to Grafana, takes a screenshot of the spike for the incident report.
Playwright
